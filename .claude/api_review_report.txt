  Phase 3 — API Convention Report
  
  Tier 1 — Breaking Changes

  T1-A: Argument ordering inconsistency in mismatch_apertures!

  ┌──────────┬───────────────────────────────────────────────────────────────────────────────────────┐
  │          │                                                                                       │
  ├──────────┼───────────────────────────────────────────────────────────────────────────────────────┤
  │ Function │ mismatch_apertures! (line 311)                                                        │
  ├──────────┼───────────────────────────────────────────────────────────────────────────────────────┤
  │ Current  │ mismatch_apertures!(mms, fixed, moving, aperture_centers, cms; normalization=:pixels) │
  ├──────────┼───────────────────────────────────────────────────────────────────────────────────────┤
  │ Proposed │ mismatch_apertures!(mms, cms, fixed, moving, aperture_centers; normalization=:pixels) │
  ├──────────┼───────────────────────────────────────────────────────────────────────────────────────┤
  │ Breaking │ Yes                                                                                   │
  └──────────┴───────────────────────────────────────────────────────────────────────────────────────┘

  The analogous in-place function mismatch! has the signature mismatch!(mm, cms, moving) — output first, storage second, data third. mismatch_apertures! breaks this pattern by placing
   cms last, after the data arguments. A user who learns mismatch! first will be surprised. By analogy with mul!(C, A, B) and other Base in-place functions, the
  destination/working-storage argument belongs early (after the output buffer).

  T1-B: mismatch(::Type{T}, fixed::AbstractArray, ...) requires explicit T for host-array dispatch (line 143)

  ┌──────────┬───────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────┐
  │          │                                                                                                                                       │
  ├──────────┼───────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────┤
  │ Function │ mismatch (method 1, line 143)                                                                                                         │
  ├──────────┼───────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────┤
  │ Current  │ mismatch(::Type{T}, fixed::AbstractArray, moving::AbstractArray, maxshift; normalization=:intensity)                                  │
  ├──────────┼───────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────┤
  │ Proposed │ Add mismatch(fixed::AbstractArray, moving::AbstractArray, maxshift; normalization=:intensity) that infers T from float(eltype(fixed)) │
  ├──────────┼───────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────┤
  │ Breaking │ No (additive) — listed here because the explicit-T method can then be kept as a documented override                                   │
  └──────────┴───────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────┘

  Calling mismatch(fixed, moving, maxshift) with plain host arrays currently hits a MethodError unless the user happens to know to pass a type argument. The CuArray dispatch (method
  2) does not require this. The inconsistency means users discover the host-array path only from documentation. Adding a no-T dispatch is non-breaking but resolves a genuine usability
   gap; grouped here because it pairs with T1-A as a "make the host-array API consistent with the CuArray API" cluster.

  Same issue applies to mismatch_apertures method 1 (line 186).

  T2-A: mismatch_apertures! has no type annotations on positional arguments (line 311)

  ┌──────────┬───────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────┐
  │          │                                                                                                                               │
  ├──────────┼───────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────┤
  │ Function │ mismatch_apertures!                                                                                                           │
  ├──────────┼───────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────┤
  │ Current  │ mismatch_apertures!(mms, fixed, moving, aperture_centers, cms; normalization=:pixels)                                         │
  ├──────────┼───────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────┤
  │ Proposed │ mismatch_apertures!(mms, cms::CMStorage{T}, fixed::CuArray, moving::CuArray, aperture_centers; normalization=:pixels) where T │
  ├──────────┼───────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────┤
  │ Breaking │ No                                                                                                                            │
  └──────────┴───────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────┘

  Every other exported function in the package has at least one typed positional argument for dispatch. mismatch_apertures! is fully untyped, so a wrong argument order produces a
  cryptic internal error rather than a helpful MethodError. This also pairs naturally with T1-A.

  T2-B: normalization Symbol not validated early (lines 256, 283–289)

  ┌──────────┬────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────┐
  │          │                                                                                                                                                                    │
  ├──────────┼────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────┤
  │ Function │ mismatch!, mismatch, mismatch_apertures, mismatch_apertures!                                                                                                       │
  ├──────────┼────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────┤
  │ Current  │ normalization = :intensity — dispatched to if normalization == :intensity … elseif normalization == :pixels … else throw(ArgumentError(...)) deep inside mismatch! │
  ├──────────┼────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────┤
  │ Proposed │ Validate at the outermost entry points (mismatch, mismatch_apertures) before any GPU work is launched                                                              │
  ├──────────┼────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────┤
  │ Breaking │ No                                                                                                                                                                 │
  └──────────┴────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────┘

  A typo like normalization=:Intensity currently passes through mismatch, uploads arrays to GPU, allocates CMStorage, runs fillfixed!, and only then throws on the GPU stream —
  expensive and confusing. Validating the symbol at entry costs nothing.

  T2-C: kwargs... in mismatch_apertures silently swallows unknown keywords (lines 192, 213)

  ┌──────────┬──────────────────────────────────────────────────────────────────────────────────────┐
  │          │                                                                                      │
  ├──────────┼──────────────────────────────────────────────────────────────────────────────────────┤
  │ Function │ mismatch_apertures (both methods)                                                    │
  ├──────────┼──────────────────────────────────────────────────────────────────────────────────────┤
  │ Current  │ kwargs... passed to CMStorage constructor, which only accepts display                │
  ├──────────┼──────────────────────────────────────────────────────────────────────────────────────┤
  │ Proposed │ Either enumerate accepted keywords explicitly, or document the valid passthrough set │
  ├──────────┼──────────────────────────────────────────────────────────────────────────────────────┤
  │ Breaking │ No                                                                                   │
  └──────────┴──────────────────────────────────────────────────────────────────────────────────────┘

  A user who passes normalization=:pixels to mismatch_apertures(::Type{T}, ...) (the host-array method, line 186–198) gets no normalization applied — it gets absorbed into kwargs...
  and forwarded to mismatch_apertures(::CuArray, ...), which does accept normalization explicitly. This works by coincidence. If the CuArray method's signature changes, the forwarding
   silently breaks.

  Tier 3 — Internal Consistency

  T3-A: display keyword silently ignores all values (lines 115–117)

  CMStorage{T}(undef, ...; display=false) — the display keyword accepts any value (including true) without warning or error. A compatibility shim that only suppresses the old behavior
   for display=false but silently ignores display=true misleads users who are trying to debug a display issue in the CPU counterpart. At minimum, display=true should emit a
  deprecation warning noting the feature is unavailable in the GPU version.

  ---
  T3-B: CuRCpair.rng is NTuple{N} with no element type (line 42)

  rng::NTuple{N}   # should be NTuple{N, UnitRange{Int}}
  The weaker type annotation means the compiler cannot specialize the closure over rng without inferring through the constructor. Tightening it is safe and non-breaking for any
  well-typed code.

  ---
  T3-C: assertsamesize checks only 3 dimensions (line 328–329)

  size(A,1)==size(B,1) && size(A,2)==size(B,2) && size(A,3)==size(B,3)
  For 1D inputs this silently passes (only dim 1 is checked against itself). For 4D+ inputs higher dimensions are ignored. Should be size(A) == size(B). This is a
  correctness/robustness issue but not a convention finding; included here because fixing it is non-breaking and low-risk.

  ---
  Overall Assessment

  The API is small and mostly coherent. The one clear modernization gap is T1-A: the mismatch! / mismatch_apertures! pair should have consistent argument ordering (storage before data
   in both). The other findings are either additive improvements or internal housekeeping. No dimension-argument or sorting-convention issues exist since the package doesn't operate
  along individual axes.

  Discussion: Modernization Policy

  Before I write the plan, a few questions:

  1. Breaking-change tolerance: The package is at v0.3.4 (pre-1.0), so breaking changes are semantically "cheap." Are you comfortable with a Tier 1 fix for T1-A (reordering
  mismatch_apertures! args), or would you prefer a deprecation shim first?
  2. Which items to act on? My recommendation: act on T1-A, T2-A (together as a cluster), T2-B, and T3-C. T2-C and T3-A/B are lower-leverage. Does that match your priorities?
  3. T1-B (no-T dispatch for host arrays): This is additive, but do you actually want users calling mismatch with plain AbstractArray arguments? Or is the intended path always through
   CuArray (and the host-array method is a convenience that should stay explicit)?
  4. Release strategy: If T1-A is approved, would you like a final v0.3.x tag before the breaking change, or go straight to v0.4.0?
